# Avatars are site-owned, seeded once from the login provider

A claimed Player's avatar is a site-owned image: seeded exactly once during the claim flow from the login provider's picture — behind an explicit "start with this picture?" confirmation, never auto-published — then managed by on-site upload, fully detached from Discord/Google thereafter. The obvious alternative, keeping avatars synced to the provider, was researched and rejected: Discord offers no HTTP webhook for user profile changes; true push requires a gateway bot with the privileged `GUILD_MEMBERS` intent, a shared guild with every user, and a third always-on service; the viable middle path (bot-token polling of `GET /users/{id}` + re-mirror on avatar-hash change) works but buys freshness for a problem the one-time-seed model deletes outright. So don't "fix" the lack of sync — it is the design.

Consequences that are deliberate: the image is always served from R2 (Discord CDN URLs 404 once a user changes their avatar — retention of old hashes is undocumented — and a dead remote URL would crash the Takumi card render, per ADR 0009's server-side-fetch constraint); uploads are acceptable despite the moderation surface because upload rights exist only behind a mod-approved Claim, making every uploader a known, accountable community member, with a mod reset-to-Medallion power as backstop; the explicit seed confirmation exists because a Google profile photo is often the person's real face, and auto-publishing it onto a public leaderboard is a privacy surprise a chosen Discord persona is not. Static images only for now; animated avatars change the pipeline and are a deliberate later decision, not an oversight.
