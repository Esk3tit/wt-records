# Takumi renders share cards, with satori as the designed fallback

Share cards (#17) are rendered in-process by Takumi (`@takumi-rs/image-response`), not by the ecosystem-default satori + resvg stack, a headless browser, or a hosted card service. Takumi is the only in-process option whose CSS engine renders the site's glass aesthetic for real — `backdrop-filter`, blend modes, and conic gradients (the nation completion ring) — where satori would force faking all three; it ships prebuilt binaries that drop into the Node/Docker image with no Chromium and no per-image cost. A 2026-07 survey of the landscape (in-process libraries, headless browsers, hosted card/screenshot services, transform CDNs) found nothing else that fits the stack without either fidelity loss, a ~300 MB browser in the container, or a recurring bill for templates that would leave the repo.

The known risk is bus factor: Takumi has a single maintainer. Two mitigations are deliberate and load-bearing: the dependency is **version-pinned**, and card templates stay within the `@vercel/og`-compatible `ImageResponse` API surface wherever possible so satori (+ `@napi-rs/canvas` — not `@resvg/resvg-js`, whose stable line has stalled since 2024) remains a cheap drop-in replacement. Glass and ring visuals would degrade under the fallback, not break. The committed static fallback card PNG is generated by script from the same template but lives in the repo precisely so the failure path never depends on the renderer working.
